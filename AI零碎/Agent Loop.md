目前主流的可以分为以下四类：
### 1. 规划与执行循环 (Plan-and-Execute)
这是对 ReAct 的重大升级，核心思想是“谋定而后动”。
- **工作机制：** 循环不再是逐个步骤地思考和行动。系统分为两个大脑：一个“规划者（Planner）”拿到总任务后，一次性生成完整的任务清单（Task 1, Task 2, Task 3...）；然后交由“执行者（Executor）”按顺序或并行去执行。执行完毕后，再由规划者检视结果，决定是否需要补充新的步骤。
- **核心优势：** 极大地缓解了长线任务中的方向迷失问题。
- **典型用途：** 长线复杂任务。例如**深度研究报告生成**（先规划章节，再分别检索，最后合成）、**复杂软件项目开发**（先写出架构设计和 TODO List，再逐个编写模块）。
### 2. 反思与批判循环 (Reflection / Critique)
这种循环不强调步骤的推进，而强调“质量的螺旋上升”。
- **工作机制：** Agent 生成初稿（代码、文章或答案）后，并不直接输出，而是将其送入一个专门的“评判节点（Critic）”。Critic 负责寻找瑕疵、报错或逻辑漏洞，并生成修改建议。Agent 拿到建议后进行重写（Refine），如此往复，直到 Critic 认为合格。
- **核心优势：** 把大模型的“幻觉”转化为自我优化的养分。
- **典型用途：** 对准确率要求极高的任务。例如**代码自动生成与 Bug 修复**（写完代码先跑单元测试，报错了就丢回给大模型分析重写）、**高级翻译与文案润色**、**数学定理证明**。
### 3. 状态机与图工作流 (State Machine / Graph-based)
随着企业级应用的深入，完全让大模型自由发挥的风险太高。因此出现了基于图（Graph）的循环控制（以 LangGraph 为代表）。
- **工作机制：** 开发者在代码层面预先定义好几个固定的“状态节点”（例如：查验用户身份、检索知识库、人工审批）和“连线”。大模型（LLM）在循环中不再决定“一切”，而是只负责在节点的分叉路口做**路由决策（Routing）**——判断下一步该走向哪条固定的管道。
- **核心优势：** 兼顾了传统软件的确定性和 AI 的灵活性，边界极强，几乎不会失控。
- **典型用途：** 企业级标准操作流程（SOP）。例如我们前面提到的 **DevOps SRE 自动排障**（必须严格按照：告警->查日志->提权限->修复 的流程走）、**智能客服**（退款流程、投诉流程必须按公司规定流转）。
### 4. 多智能体编排循环 (Multi-Agent Orchestration)
当单个 Agent 的提示词太长、塞不下所有工具时，循环就变成了团队协作。
- **工作机制：** 存在一个“主管（Orchestrator）”Agent。循环的第一步是主管对任务进行拆解，然后将其分发给下属的专业 Agent（比如数据分析 Agent、绘图 Agent）。下属们各自执行自己的小循环，完成后向主管汇报。如果结果不匹配，主管会打回重做。
- **核心优势：** 实现了能力解耦，每个小 Agent 只需要专注于一件事，大幅降低出错率。
- **典型用途：** 跨领域的复合型任务。例如**自动化金融研报生成**（数据采集 Agent 看财报、模型 Agent 算估值、文案 Agent 写报告）、**沉浸式游戏 NPC 社会模拟**。

**总结对比：**

| **Agent Loop 类别**    | **核心特征**      | **适用场景**     | **就像人类的...** |
| -------------------- | ------------- | ------------ | ------------ |
| **ReAct**            | 走一步看一步        | 简单查询、单步工具调用  | 日常即兴对话       |
| **Plan-and-Execute** | 先列清单，再逐个打钩    | 长期研究、复杂项目开发  | 项目经理做甘特图     |
| **Reflection**       | 不断自我挑刺和修改     | 代码编写、内容润色    | 作家反复修改草稿     |
| **State Machine**    | 按规定路线走，遇岔路再思考 | 企业 SOP、自动化客服 | 按照操作手册干活     |
