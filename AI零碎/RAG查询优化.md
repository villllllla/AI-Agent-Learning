在检索增强生成（RAG）系统中，用户输入的原始问题往往存在表述模糊、缺乏上下文或过于复杂等问题。直接拿这些原始问题去向量数据库里进行相似度检索，往往效果很差。
查询优化（Query Optimization）**或**查询转换（Query Translation/Transformation）就是充当用户和检索系统之间的“翻译官”和“调度员”。

RAG的查询优化主要分为三大流派：**查询转换**、**查询路由**和**查询结构化**。
### 1. 查询转换 (Query Transformation)
这是最常见的一类优化方式，核心思想是**让LLM先对问题进行“加工”，再去检索**。
- **查询重写 (Query Rewriting)：** 用户的输入可能有拼写错误，或者省略了主语。LLM 会结合历史对话记录，将当前问题补全。例如，用户问“它明天的天气怎么样？”，系统会将其重写为“北京2026年5月22日的天气怎么样？”。
- **多查询扩展 (Multi-Query Expansion)：** 同一个意思有不同的表达方式，由于向量检索依赖字面或语义的相似度，单一查询容易漏掉相关文档。此时，LLM 会将原问题改写成 3-5 个不同角度的相似问题，分别进行检索，最后对结果取并集并进行重排（Rerank）。
- **查询分解 (Query Decomposition / Sub-queries)：** 当面对复杂问题时（比如“对比A公司和B公司2023年的财报利润”），LLM 会将其拆解为多个子问题：“A公司2023年利润是多少？”和“B公司2023年利润是多少？”，分别检索后再由大模型汇总。
- **退一步提示 (Step-back Prompting)：** 当问题过于具体时，系统会生成一个更宏观、更抽象的“退一步”问题。比如用户问“为什么昨天发布的XX型号手机电池耗电快？”，退一步问题可能是“智能手机电池耗电快的一般原因是什么？”。同时检索这两个问题，能为最终回答提供更丰富的背景知识。
- **HyDE (Hypothetical Document Embeddings - 假设性文档嵌入)：** 这是一种“无中生有”的策略。面对用户的问题，LLM 先凭借自身的预训练知识“瞎编”或者预测一个答案（假设性文档）。然后，系统不拿原问题去检索，而是拿这个生成的答案去向量库里寻找与其最相似的真实文档。因为答案的语义特征往往比短小的问题更容易匹配到目标文档。
#### 实现方式
**核心就是通过精心设计的 Prompt 让 LLM 进行加工**，但在工程实现上，这不仅是一句 Prompt，而是一个完整的“提示词 + 结构化输出解析 + 循环检索”的代码流水线（Pipeline）。
以多查询扩展（Multi-Query）为例，具体的实现步骤如下：
**第一步：定义 Prompt，让 LLM 吐出结构化数据** 你不能只让大模型随便回答，必须要求它按特定格式（如 JSON 或特定分隔符）输出，以便代码后续处理。

```Plaintext
你是一个专业的AI检索助手。你的任务是基于用户的原始问题，生成3个不同表达方式的相似问题，以便在向量数据库中进行更全面的检索。
请从不同的语义角度进行改写，但保持核心意图不变。

原始问题：{user_query}

请严格按以下JSON格式输出，不要包含任何其他解释性文字：
{
  "queries": [
    "改写问题1",
    "改写问题2",
    "改写问题3"
  ]
}
```

**第二步：代码解析与并发检索** 在你的代码（比如使用 LangChain 或 LlamaIndex 框架）中，拿到 LLM 返回的 JSON 后，提取出这 3 个问题。然后，代码会开启多线程，分别拿这 3 个问题加上原问题（共 4 个），去向量数据库里执行 similarity_search。
**第三步：去重与重排（Rerank）** 每个问题可能都搜到了 5 块文档片段，总共 20 块片段。代码会将它们合并去重，并交给一个轻量级的排序模型（如 BGE-Reranker），挑出最相关的 Top 5，最终喂给大模型生成答案。

### 2. 查询路由 (Query Routing)
很多企业的知识不仅存在向量数据库里，还存在关系型数据库（MySQL）或图数据库（Neo4j）中。查询路由的作用是**判断问题该去哪里找答案**。
- **逻辑路由 (Logical Routing)：** 基于规则或 LLM 判断问题的意图。如果是问具体指标（“上个月销量多少”），路由到 SQL 数据库；如果问概念解释（“什么是RAG”），路由到向量数据库；如果是问实体关系（“张三和这家公司的关系”），路由到图数据库。
- **语义路由 (Semantic Routing)：** 根据问题语义，选择最合适的 Prompt 模板或特定领域的索引库（比如将医疗问题导向医疗知识库，法律问题导向法律知识库）。
### 3. 查询结构化 (Query Structuring)
当用户的自然语言中包含明确的筛选条件时，纯粹的语义检索（也就是只靠向量计算）会失效。结构化查询的作用就是**提取硬性过滤条件**。
- **自查询检索 (Self-Query Retrieval)：** 将自然语言转换为“查询向量 + 元数据过滤器（Metadata Filters）”。比如用户问：“帮我找一篇2023年发表的关于Transformer的论文”。系统会将其结构化为：
    - **检索语义：** "关于Transformer的论文"
    - **过滤条件：** {"year": {"$eq": 2023}}
- **Text-to-SQL / Text-to-Cypher：** 直接将自然语言问题转化为 SQL 语句或图数据库的查询语言（Cypher），从而在结构化数据源中进行精确查询。
> **核心原则：** 没有一种优化方法是万能的。现在的复杂 RAG 系统（如 Advanced RAG）通常会将上述策略组合使用。比如：先通过**路由**判断要去哪个库，再用**多查询扩展**增加命中率，最后利用**重排（Reranker）**剔除不相关的噪音。
