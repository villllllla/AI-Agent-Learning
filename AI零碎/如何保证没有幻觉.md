### 什么是幻觉
幻觉就是LLM一本正经的说了一堆看似正确的话，但其实压根不存在。出现的原因就是它只是在预测下一个词，当没有可以可靠依据时，就会拼接出一个看起来合理的答案，但内容可能是错的。


Agent 主要通过以下 5 种核心工作流来保证内容的真实性：
### 一、外部工具调用 (Tool Calling)
单纯的 LLM 在做数学题或查实时天气时容易瞎编，因为它的知识停留在训练截止日，且本身并不擅长精确的数理计算。
- **Agent 的做法：** **赋予模型调用外部工具的能力**。遇到算术题，Agent 会写一段 Python 代码或调用计算器 API 去算；遇到最新新闻，它会调用 Google 搜索 API 去查；遇到查询订单，它会直接调取公司的 SQL 数据库。
- **防幻觉原理：** **把需要“绝对精确”的任务剥离出来**，交给传统的确定性系统，LLM 只负责“理解意图”和“组装最终答案”。
#### 工具调用幻觉
在实际开发中，Agent 在调用工具时最容易出现以下三种幻觉：
1. **“无中生有”**：模型捏造了一个根本不存在的工具名称（比如你只给了 `search_db`，它硬造了一个 `delete_db`）。
2. **“张冠李戴”**：选错了工具（比如该用 `get_user_info`，它却调用了 `get_order_history`）。
3. **“胡编参数”**：工具选对了，但传入的 JSON 参数格式错误，或者捏造了不存在的参数值。
要彻底压制这种“工具幻觉”，业内目前有一套非常成熟的工程打法，主要分为**设计期预防**、**生成期约束**和**执行期纠错**三个层面：
##### 1. 预防期：为工具编写“防御性说明” (Schema Engineering)
大模型是靠阅读工具的描述（Description）来决定用哪个工具的。**绝大多数选错工具的幻觉，是因为开发者给的工具描述太模糊。**
- **加入反向提示（Negative Prompts）：** 不仅要告诉模型这个工具“能做什么”，还要明确指出“不能做什么”以及“容易和谁混淆”。
    - _反面教材：_ `用来查询用户信息。`
    - _最佳实践：_ `通过用户 ID 查询用户的基本信息（姓名、邮箱）。注意：如果你需要查询用户的购买记录，请绝对不要使用此工具，而是使用 get_order_history 工具。`
- **参数强类型限制：** 尽量少用自由文本字符串。能用枚举（Enum）的就必须用枚举。例如，让它传状态码，不要写 `status: string`，而是写 `status: ["running", "stopped", "failed"]`。、
##### 2. 调度层：动态工具检索 (Tool RAG)
如果你给一个 Agent 同时挂载 50 个工具，模型的注意力机制就会崩溃，选错工具的概率呈指数级上升。
改进方案： 不要一次性把所有工具塞给模型。引入一个轻量级的路由机制：当用户提问时，先通过向量检索（RAG），从工具库里匹配出与当前任务最相关的 3 到 5 个工具，然后再把这几个工具交给 Agent 去做决策。
效果： 极大地缩小了模型的选择范围，从源头上切断了“张冠李戴”的可能。

##### 3. 生成期：强制结构化输出 (Constrained Decoding)
为了防止模型胡编参数、少传必填项或者输出非法的 JSON，现在业内普遍采用受限解码技术。
改进方案： 在推理引擎底层拦截：
- 使用像 OpenAI 的 Structured Outputs（结构化输出）特性，或者在开源模型中使用 Outlines、Instructor 这样的库。它们在底层通过干预模型的词表概率，物理级地强制模型只能输出符合你预设 JSON Schema 的结构。如果模型想生成一个 Schema 之外的 key，这个生成的概率会被直接降为 0。
- 云端结构化输出：
	- **OpenAI Structured Outputs：** OpenAI 在 2024 年下半年正式在 API 中内置了原生的受限解码能力。你在发起请求时，传入 `response_format: { type: "json_schema", strict: true }`，OpenAI 的服务器会在底层构建 FSM 来保证 100% 遵守你定义的 Schema。
	- **Instructor 库：** 这是调用 API 时的标配封装库。它实际上是“柔性”约束：它先告诉模型要什么结构，如果模型弄错了，Instructor 会自动抓取 Pydantic 的报错信息，并在后台自动重新请求（Retry），直到模型改对为止。

##### 4. 执行期：闭环报错与自我反思 (Self-Correction Loop)
哪怕前面做得再好，模型偶尔还是会传错一个参数。关键在于出错后不能让程序直接崩溃。
改进方案： 在代码逻辑里加一层拦截器。当工具执行失败（比如因为参数不全抛出了 KeyError 或 API 报错）时，不要把错误直接抛给最终用户。
如何做： 把系统原生的报错信息抓取下来，拼成一段话反馈给 Agent 自身：“你刚才调用的工具失败了，系统报错说缺少 user_id 参数。请你重新检查你的参数，并再试一次。”
效果： 现代大模型（如 GPT-4o, Claude 3.5 Sonnet）拥有极强的纠错能力，只要把真实的报错日志扔给它，它通常能在第二轮或第三轮自己把参数改对。

##### 5. 架构层：职责拆分 (Multi-Agent 分工)
如果一个 Agent 既要查数据库，又要写文件，还要发邮件，它的角色就会精神分裂，导致工具误用。
改进方案： 拆分 Agent。比如，专门做一个“数据库查询 Agent”，它手里只有 SELECT 工具；再做一个“邮件发送 Agent”，手里只有 SMTP 工具。主控 Agent 只需要用自然语言将任务分配给它们即可。
核心思路总结： 不要盲目相信大模型的自觉性。把工具当成一个极其严格的 API 接口来设计，用机制去限制它，用报错去引导它。

### 二、检索增强生成 (RAG, Retrieval-Augmented Generation)
当被问到垂直领域（如公司内部财报、特定医疗指南）的问题时，模型如果没有学过，就会凭空捏造。
- **Agent 的做法：** 在回答前，Agent 会先去外挂的**私有知识库**里搜索相关的文档片段，然后把这些片段连同用户的问题一起丢给模型，并下达严厉的系统指令：“只能基于提供的文档回答，如果文档里没有，必须回答‘我不知道’。”
- **防幻觉原理：** 把“闭卷考试”变成“开卷考试”，实现**事实溯源 (Grounding)**。答案的每一句话都可以追溯到具体的参考文档。
[# 滴滴Agent岗二面：如何规避 RAG 系统中大模型的幻觉？](https://mp.weixin.qq.com/s/woFAkWRno6OxstLFzqm6aA)
#### RAG产生幻觉的原因
##### 第一种：检索层没找到，LLM 靠「猜」回答
知识库中都是不相关的信息，大模型没有充足的语料，只能瞎猜
具体例子：你的知识库里记录的退款政策是「7 天无理由退款」，但某次检索没召回到这个 chunk，LLM 用自己的「知识」给出了「30 天退款」的答案，用户按这个信息去操作，结果退款失败，这就是检索层幻觉造成的实际损失。
##### 第二种：检索到了，但 LLM 没有严格照着说
在知识库中检索到了对应的片段，但是大模型以资料为基础，增加了自己的推断、补充了原文没有的细节、甚至将不想关的信息混在一起拼接出更完整的答案，读者很难判断哪句来自文档，哪句来自LLM。
![](AI-Agent-Learning/AI零碎/assets/如何保证没有幻觉/file-20260521064802763.jpg)

#### 解决方案
##### 方案一：用 Prompt 告诉 LLM「只准照着资料说」
在 system prompt 里明确立规矩，只能用资料里的内容、资料里没有就说不知道、不准推断和补充。这几条规则，每一条都有它的道理：
```
回答规则：
# 让 LLM「知道自己的边界」，明确告诉它，你只是个传话人，不是百科全书，只能照资料说话。
1. 只能使用【参考资料】中的信息来回答，不得引入资料之外的知识
   
# 给了 LLM 一个「合法的逃生出口」，当资料里没有答案时，说「不知道」比编一个看起来合理的答案要好得多，这条规则允许 LLM 优雅地认怂。
2. 如果参考资料中没有足够信息，必须明确回答：「根据现有资料，无法回答该问题」
   
# 让 LLM 在回答时主动对照原文，相当于给它加了一个自检动作，哪句话来自哪条资料必须说清楚。
3. 回答时标注信息来源（来自哪条参考资料）

4. 不要推断、不要猜测、不要补充资料中没有明确说明的内容
```

##### 方案二：检索质量差时，直接拒绝回答
方案一解决了「LLM 超发挥」的问题，但如果检索这一步就失败了，prompt 里全是无关内容，光靠 Prompt 约束是拦不住的。那怎么办？与其让 LLM 在低质量的上下文里强行生成一个「可能是错的答案」，不如直接告诉用户「知识库里没有这个信息」。前者让用户误以为答案是对的，后者虽然看起来不那么智能，但至少不会给用户错误信息。
怎么判断检索质量够不够好？用 Rerank 模型打分。整个流程很直接：检索召回 top-K 个候选 chunk，Rerank 模型对每个候选打一个 0 到 1 的相关度分数，然后看最高分的值。如果最高分低于阈值，说明这次检索根本没找到有用的内容，直接返回「知识库无相关信息，建议联系人工」，不让 LLM 强行回答；如果最高分达标，就把所有达标的 chunk 筛出来喂给 LLM。

这个阈值到底设多少？没有通用数字，得按你自己的业务数据调。调法是拿一批「已知答案在知识库里」和「已知答案不在知识库里」的测试集跑一遍，看 Rerank 分数分布，找那个能把两类分开的切点。一般经验值在 0.3 到 0.6 之间：对精度要求高（金融、医疗）偏高，对召回率宽松（闲聊型问答）偏低。先从中间值开始，根据线上误判的样例往上或往下调。

方案二负责「检索质量差时不让 LLM 胡说」，方案一负责「检索质量还行但 LLM 容易超发挥时把它框住」。这两个是 RAG 系统上线前的基础配置，成本也最低。

##### 方案三：答案生成后，逐条核查有没有「凭空捏造」
LLM 生成完答案，再用另一个 LLM（或者同一个 LLM）回头检查，答案里的每一条关键信息，在 chunk 里有没有对应的依据。没有依据的内容，标注「无法核实」或者直接删掉。实现方式是发起「第二次 LLM 调用」，把原始答案和 chunk 一起送给 LLM，让它充当编辑角色，逐条核查每个声明有没有来源支撑。就像文章写完还要请编辑审阅一遍，方案一是事前约束作者的行为，方案三是事后审稿。
**代价**：多一次 LLM 调用，响应延迟会增加，成本也会翻倍。所以一般只在对准确性要求极高的场景使用，比如医疗问诊、法律咨询，答错了代价很大的地方。普通的企业知识库问答，方案一 + 方案二组合通常已经够用。

##### 方案四：强制 LLM 输出时带上来源编号
如果说方案三是在内容层面做校验，那方案四就是在格式层面做约束，和方案一的「Prompt 约束」配合使用效果更好。让 LLM 输出结构化的 JSON，每个结论都必须填写来自哪条参考资料的编号。这样一来，用户看到答案的同时就能看到来源，系统也可以自动验证每条结论的来源 ID 是不是真实存在。LLM 输出的格式大致是这样：

```JSON
{  
 "answer": "完整回答",  
 "statements": [  
     {"claim": "具体结论1", "source_ids": [1, 2]},  
     {"claim": "具体结论2", "source_ids": [3]}  
 ],  
 "confidence": "high/medium/low"  
}
```

这个方案为什么有效？
原因有两个：一是 LLM 在构建 JSON 时必须主动想「这条结论我从哪条资料里找到的」，这个过程本身就会减少瞎编的概率，就好比你让学生写论文必须标注参考文献，他自然就不敢随便编了；二是系统拿到 JSON 后可以做程序化验证，`source_ids` 里的编号如果对应的 chunk 和 `claim` 明显不相关，就可以自动过滤掉这条结论。

##### 方案组合
从易到难，按场景递进来看：普通的企业知识库和客服问答，方案一加方案二是必须做的基础配置，成本极低，幻觉问题能解决大半；对可信度要求高的场景，比如金融分析、法律文档，再加上方案四，让每条答案都带上来源编号，用户可以自己去原文核实；对准确性要求极高、容错率极低的场景，比如医疗问诊、合规审核，答错了代价很大，这时候四个方案全上，在延迟和成本上付出代价，换取最高级别的准确性保证。

### 三、反思与自我纠错 (Self-Reflection & Critique)
基础大模型生成内容通常是“脱口而出”，不加思索。
- **Agent 的做法：** 在工作流中硬性植入验证环节。Agent 生成初步答案后，并不直接输出给用户，而是触发一个“自我审查”流程（有时会调用另一个专门负责找茬的“裁判 Agent”），问自己：“这个答案符合逻辑吗？有数据支撑吗？引用的链接真实存在吗？” 如果发现问题，它会自己推翻重做。
- **防幻觉原理：** 用多轮迭代和过滤机制拦截荒谬的输出。


### 四、思维链规划 (Chain of Thought, CoT)
如果直接让模型给最终答案，它很容易在长逻辑链条中迷失并开始胡言乱语。
- **Agent 的做法：** 强迫模型在行动前先写下计划，把大问题拆解成小步骤（例如：1. 先搜索X的定义 -> 2. 对比Y的数据 -> 3. 提取关键差异 -> 4. 总结）。
- **防幻觉原理：** 逻辑的透明化不仅能大幅提升推理准确率，也能防止模型在解决复杂问题时发生“认知跳跃”。

### 五、多智能体协作与辩论 (Multi-Agent Debate)
单一模型容易陷入自己的逻辑死胡同或证实偏差。
- **Agent 的做法：** 让多个 Agent 扮演不同的专业角色。比如写代码时，一个 Agent 负责写（程序员），一个负责测（测试员），一个负责查漏洞（安全员）。程序员写出方案后，测试员去挑刺，两者不断交互修改，直到测试通过。
- **防幻觉原理：** 利用“兼听则明”的交叉验证，不同视角的 Agent 相互制约，打破单一模型的盲区。

**总结来说：** 单体大模型对抗幻觉靠的是“死记硬背”（扩大参数量、清洗训练数据），而 Agent 对抗幻觉靠的是“科学的做事流程”——不懂就去搜，算不清就用工具，写完之后再检查一遍。
