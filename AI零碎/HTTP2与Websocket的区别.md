#### 1. 核心通信模型不同（最关键）
- **[[HTTP/HTTP2]] (请求-响应模型)：** 依然是半双工的哲学。**必须由客户端（浏览器）先发号施令（发请求），服务器才能回答（给响应）。** 虽然有 SSE 可以让服务器持续推数据，但前提也是客户端先发了个请求建立监听。
- **WebSocket (全双工模型)：** 建立连接后，通道是平等的双向车道。**服务器可以随时随地、主动向客户端推送消息**，不需要客户端先问它。
#### 2. 报文开销与轻量度
- **HTTP/2：** 哪怕压缩了头部，它依然承载着 HTTP 的语义（Method、Status Code、Headers 等），适合传输文档、图片、API 数据。
- **WebSocket：** 在完成最初的握手（借用 HTTP 协议）之后，它就蜕变成了一个极简的二进制协议。一个 WebSocket 帧的头部**最小只有 2 个字节**，没有多余的 HTTP 语义，极其轻量。

#### 3. 适用场景的区别

|**特性维度**|**HTTP/2 (+ SSE)**|**WebSocket**|
|---|---|---|
|**通信方向**|客户端请求 -> 服务器流式响应|真正的双向实时通信|
|**主打场景**|网页秒开、RESTful API 提速、**大模型对话 (ChatGPT 的打字机效果)**|**多人联机游戏、实时聊天室、协同编辑 (飞书文档)**|
|**状态维持**|无状态（靠每次请求带 Token/Cookie）|有状态（长连接期间身份一直保持）|
|**服务器主动性**|弱（依靠客户端的长轮询或 SSE 监听）|强（可以主动向任意在线客户端发消息）|
