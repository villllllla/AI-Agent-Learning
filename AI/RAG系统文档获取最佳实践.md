在企业级 RAG（检索增强生成）系统的设计中，**“文档获取”**（通常称为数据摄取 Data Ingestion 或 ETL 流水线）是决定系统最终效果的基石。行业内常说 _“Garbage in, garbage out”_，如果文档解析和获取做得不好，后端的大模型能力再强也无法生成准确的答案。

针对企业场景，文档获取不仅要解决“怎么拿数据”的问题，还要解决“怎么解析复杂格式”、“怎么保持语义完整”以及“怎么管理权限”等问题。以下是针对企业 RAG 文档获取环节的推荐方案和业界最佳实践：

---

### 1. 数据接入层 (Data Connection)：多源数据的自动化采集

企业内部的数据通常散落在不同的系统中（如 SharePoint, Confluence, Jira, 飞书/钉钉知识库, ERP 数据库等）。

- **推荐方案：使用标准化连接器 (Connectors)**
    
    - **开源生态：** 推荐使用 **LlamaIndex Data Connectors (LlamaHub)** 或 **LangChain Document Loaders**。它们提供了数百个开箱即用的连接器，可以直接拉取 Notion、Confluence、飞书文档等 API 数据。
        
    - **低代码/ETL工具：** 对于需要定时同步的大规模企业数据，推荐引入 **Airbyte** 或 **Apache SeaTunnel** 等数据集成工具。它们能够稳定地管理全量与增量同步，将各业务系统的数据统一抽取到 RAG 的原始数据湖或临时存储中。
        
    - **增量更新机制：** 企业文档是动态变化的。建议通过监听业务系统的 Webhook 或使用 CDC（变更数据捕获）技术，实现文档的**实时或准实时增量更新**（新增、修改、删除），避免每次都全量重建向量库。
        

### 2. 文档解析层 (Document Parsing)：复杂格式的“解构”

这是目前 RAG 系统最大的痛点。企业文档多为 PDF、PPT、扫描件或包含复杂表格和多栏排版的 Word 文档。简单的文本提取会导致段落错位、表格混乱。

- **推荐方案：引入具备“版面分析（Layout Analysis）”的解析工具**
    
    - **全能型开源工具（瑞士军刀）：`Unstructured`** 它能深入理解文档格式，自动识别标题、段落、列表、表格和图像。支持数十种文件格式，是目前最通用的开源解析引擎。
        
    - **针对复杂 PDF/科研文献：`Docling` 或 `MinerU`**
        
        - _Docling_（由 IBM 开源）在处理带有复杂表格、多栏排版和公式的 PDF 时表现极佳。
            
        - _MinerU (PDF-Extract-Kit)_ 是国内开源的优秀工具，对中文排版、数学公式和阅读顺序的还原度非常高。
            
    - **商业级/API 方案（追求极致效果）：** 如果预算充足，推荐使用专门为 RAG 优化的商业解析 API，如 **LlamaParse**（对复杂表格转化为 Markdown 支持极好）、**Azure Document Intelligence** 或 **阿里云百炼文档解析**。它们底层通常结合了多模态大模型（VLM）和 OCR 技术。
        
    - **图表处理策略：** 遇到文档中的图片或图表，推荐调用多模态大模型（如 Gemini Pro Vision 或 GPT-4o）生成详细的图像描述（Caption），将生成的文本与原文档一起向量化。
        

### 3. 智能切分与重组 (Chunking Strategy)：保留上下文语义

传统的“按固定字数（如 500 Token）切分”会把一个完整的语义生生截断，导致大模型回答缺少上下文。

- **推荐方案：语义切分与高级重组策略**
    
    - **按结构/语义切分：** 基于上一步的解析结果，按“标题层级（Heading）”、“段落（Paragraph）”或“Markdown 结构”进行自然切分，确保每一块的语义完整。
        
    - **父子文档切分 (Parent-Child Chunking / Small-to-Big)：** 这是目前企业 RAG 最推荐的策略。将文档切分为大块（父文档，如 1000 Token），再将大块细分为小块（子文档，如 200 Token）。**向量化和检索时使用小块**（精度高，命中率高），**喂给大模型生成时提供对应的父文档**（上下文丰富，防止幻觉）。
        
    - **摘要索引 (Summary Indexing)：** 为长文档或长片段让大模型先生成一段全局摘要。检索时先检索摘要，命中后再拉取全文。
        

### 4. 元数据增强 (Metadata Enrichment)：提升检索精度的杀手锏

在将切分好的文本（Chunks）存入向量数据库之前，**必须**为其打上丰富的元数据标签。纯向量检索（Dense Retrieval）往往不够准，企业级应用高度依赖带有元数据过滤的混合检索（Hybrid Search）。

- **推荐提取的元数据维度：**
    
    - **基础信息：** 文档标题、作者、创建时间、最后更新时间、文档来源 URL。
        
    - **结构信息：** 当前 chunk 属于哪个章节（如 `章: 财务报表, 节: 2025Q3营收`）、页码。
        
    - **业务标签：** 所属部门、文档类别（合同、技术规范、HR 政策等）。可以通过小模型（如 NLP 分类器）或大模型在入库前自动打标。
        

### 5. 权限与安全管理 (Security & Access Control)

在企业环境中，CEO 能看到的报表与普通员工能看到的规范是不同的。RAG 系统绝不能越权泄露文档。

- **推荐方案：文档级权限控制 (Document-level Security)**
    
    - 在获取文档时，**同步提取该文档的 ACL（访问控制列表）或权限组标签**，并将其作为元数据（Metadata）与文档 Chunk 一起存入向量数据库。
        
    - 当用户发起提问时，RAG 系统首先获取该用户的身份和权限标签。
        
    - 在向量数据库执行检索时，**强制加入元数据过滤条件**（例如 `WHERE 权限组 IN ('公共', '销售部')`），确保检索召回的内容本身就是该用户有权查看的，从物理层面上杜绝越权回答。
        

### 总结：企业级数据获取流水线参考架构

一个高标准的文档获取流通常长这样：

1. **触发：** 监听系统 Webhook 发现新文件（如一份几十页的产品白皮书 PDF）。
    
2. **解析：** 调用 `MinerU` 或 `LlamaParse`，将其转化为带有清晰层级的 Markdown，同时提取出表格内容。
    
3. **清洗：** 去除页眉、页脚、水印、无效空白。
    
4. **切片：** 采用“父子文档”策略，按 Markdown 标题进行语义切分。
    
5. **增强：** 让 LLM 快速总结各段落摘要，提取时间、作者、权限等注入 Metadata。
    
6. **入库：** 计算向量嵌入（Embedding），连同文本和元数据存入支持稀疏向量的现代向量数据库（如 Milvus, Qdrant 或 Elasticsearch）。
