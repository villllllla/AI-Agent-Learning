### 大模型是什么
大模型(LLM, Large Language Model)是指:
- 参数大：模型内部有数千个参数，可以理解为记住了海量知识的精华
	- 设大模型是一个极其庞大的高级音乐调音台。调音台上密密麻麻布满了无数个**小推子和旋钮**，这些旋钮就是“参数”。当一个问题（声音）输入进来时，电流会穿过这无数个旋钮，最终输出一个回答。在“训练”大模型的过程中，科学家其实就是在不断微调这几十亿、几百亿个旋钮的位置，直到它输出的回答听起来最聪明、最准确为止。
- 训练数据大：训练人员给他”喂了“海量的知识，如互联网上的文章、知识、新闻、论坛内容等大量内容。
大模型训练，就是一个通过不断试错和统计计算（损失函数 (Loss Function) 与 前向传播 (Forward Pass)），寻找一组“最优参数”（梯度下降 (Gradient Descent) 与 反向传播 (Backpropagation)），使得模型预测的误差降到最低的数学优化过程。

### 什么是Token
Token是指大模型处理文本的”基本单元“，既不是一个字，也不是一个词，可以理解为把”文字“切碎之后的小碎片。


#### Token切分：
「我今天心情很好」这句话,大模型不是一个字一个字地读,而是先把它切成若干个 Token,比如可能
切成:「我」「今天」「心情」「很好」,每个碎片就是一个 Token。
英文的切法更直观:＂Hello world＂通常会被切成\[＂Hello＂,＂world＂\]，1 个英文 Token 大约等于 0.75 个英文单词；1 个汉字通常占用 1 到 2 个 Token。

#### 切分的依据：
大模型首先会将大规模语料，通过统计算法（最流行的是 **BPE算法（Byte Pair Encoding，字节对编码）**，类似于哈夫曼编码（Huffman Coding），核心思想是：**把经常紧挨着出现的字符，合并成一个新的大字符**。），训练生成词表，Tokenizer根据词表将文本切分为Token。
完整流程为：文本输入 -> **Tokenizer 切分成一个个 Token** -> **把 Token 映射成对应的 ID (数字)** -> 喂给大模型计算 -> 输出一堆新的 Token ID -> **Tokenizer 把 ID 解码回人类文本**。

#### Token的重要性：
1. 决定模型能处理的文本长度上限:每个大模型都有＂最多能处理多少 Token＂的上限
2. 决定API调用费用：调用大模型的API是按 Token 计费的，你发给它的文字越长、它回复的越多，花的钱越多

#### Token大小
##### 网络传输与 CPU 内存
在分词器（Tokenizer）把文本切成 Token 后，每个 Token 都会被映射成一个整数 ID（比如 `Token ID: 5234`）。
在代码里，这个 ID 通常用一个 32位整型 (`int32`) 或 64位整型 (`int64`) 来存储。
- 如果按 `int32` (4 字节) 计算，128,000 个 Token 的体积是：
    $128,000 \times 4 \text{ Bytes} \approx 512 \text{ KB}$
- **结论**：在你的 Web 后端发送 HTTP 请求，或者在网络传输时，128K 上下文的数据包只有不到 1MB，微不足道。

##### 推理时的 GPU 显存（KV Cache）
当这 128,000 个 ID 进入大模型内部进行推理时，模型并不是只记住 ID 就行了。为了能结合上文预测下一个词，模型需要在每一层神经网络中，把每个 Token 的“特征状态”缓存下来。这个缓存区在 AI 领域叫 **KV Cache (Key-Value Cache)**。
随着 Token 数量的增加，KV Cache 占用的显存是**线性爆炸**的。
- _以常见的 Llama-3 8B 模型为例（16位精度，fp16，每个数字占 2 字节）_：
    保存 1 个 Token 的状态大约需要消耗 $0.125 \text{ MB}$ 的显存。
    如果你塞满了 128,000 个 Token，仅仅是为了“记住”这些上下文，就需要消耗掉：
    $128,000 \times 0.125 \text{ MB} \approx \mathbf{16 \text{ GB 显存 (VRAM)}}$
- **结论**：128K Token 在物理层面，意味着系统要在极其昂贵的 GPU 显存里，硬生生切出十几 G 的空间来存放它的上下文状态矩阵。这也是为什么现在的 MLOps 都在搞 `vLLM`、`PagedAttention` 技术，目的就是像操作系统管理虚拟内存分页一样，去优化这段显存。

#### Token 转化链路
##### 阶段 1：Agent 端（后端服务）—— “只算账，不真转”
1. **接收输入**：你的 Java/Python 后端（Agent）收到前端传来的用户提问（Text）。
2. **本地预计算（Pre-flight Check）**：你的后端代码会引入一个**本地的 Tokenizer 库**（比如 OpenAI 的 `tiktoken` 或者是 HuggingFace 的 `transformers` 库，加载在内存中，大约为几十 MB，构建为一个高效的字典树（Trie Tree）或 HashMap）。
3. **计算 Token 数量**：后端将组装好的那一大坨文本（系统提示词 + 历史记录 + RAG 文档 + 用户提问）喂给本地的 Tokenizer。Tokenizer 瞬间将其转换为 ID 数组：`[12, 453, 8999...]`。
4. **【关键点】丢弃 ID，保留文本**：你的后端拿到这个 ID 数组后，**只做一件事：调用 `length()` 或 `size()` 函数算一下总数！** 如果总数是 3900，小于 4000 的窗口限制，OK，安全。然后，你的后端会**把这个 ID 数组直接扔掉**。
5. **发送 HTTP 请求**：你的后端把最原始的**纯文本字符串**包装进 JSON 中，通过 HTTP/HTTPS 发送给大模型厂商的 API Gateway。
##### 阶段 2：LLM 端（大模型厂商的服务器）—— “真刀真枪的转化”
1. **接收请求**：厂商的 API Gateway 收到你发来的 JSON（里面全是纯文本）。
2. **服务端 Tokenization**：厂商的服务器内部，会运行一个和你的本地库完全一模一样的 Tokenizer。它把收到的文本**真正地切分并转换为 Token ID 数组**。
3. **进入 GPU**：这个由厂商服务器生成的 ID 数组，会被立刻送入 Embedding 层，映射成浮点数向量（`ID 5234` $\rightarrow$ `[0.12, -0.45, 0.89, 0.01, ... 还有 4092 个数字]`），最后进入 GPU 的 KV Cache 进行推理。
4. **返回结果**：GPU 吐出一串新的 Token ID（代表回答），厂商服务器的 Tokenizer 再把这些 ID **解码（Decode）回人类可读的纯文本**，最后通过 HTTP 响应（通常是流式 SSE）发回给你的 Agent Backend。

#### 相关问题
##### 1. 推理训练时，是对 Token 的 ID 值进行计算吗？
Token ID 只是一个**离散的类别标签**，就像数据库表里的主键 `Primary Key`。
假设 `苹果 = ID 10`，`香蕉 = ID 20`。如果模型直接用 ID 拿去做加减乘除计算，就会得出荒谬的结论（难道香蕉是苹果的两倍？）。
那么模型到底是怎么计算的呢？这中间有一个极其关键的桥梁：**Embedding 层（词嵌入映射）**。
你可以把 Embedding 层理解为大模型内部的一个极其庞大的 **HashMap / 字典表**：
1. **查表**：当模型拿到 `ID: 5234` 时，它的第一步根本不是计算，而是去这个大字典里做一次 Look-up（查表）。
2. **转为向量**：查表的结果，是一串长长的浮点数数组（这个数组就是 Vector 向量，通常有 4096 个维度）。
    - 例如：`ID 5234` $\rightarrow$ `[0.12, -0.45, 0.89, 0.01, ... 还有 4092 个数字]`。
3. **矩阵运算**：模型后续所有的乘法、加法、概率统计、注意力机制（Attention），**全部都是在这些成千上万维的浮点数向量上进行的**。因为只有浮点数向量，才能表达出“语义的远近”（比如苹果和香蕉的向量在数学空间里距离很近）。
##### 2. 为什么在Agent不直接传输 ID给大模型
1) 幻觉 1：传 ID 真的能减少传输长度吗？（JSON 的诅咒）
	这是一个极度“反常识”的数据。大部分大模型 API 使用的是 HTTP + JSON 协议。
	- **传纯文本**：比如英文单词 `apple`，纯文本占 **5 个字节**。
	- **传 Token ID**：`apple` 的 ID 可能是 `16113`。在 JSON 数组中，它会被序列化为字符串 `[16113]`，这就占了 **7 个字节**。
	- **结论**：在基于 JSON 的 RESTful API 中，**传一长串数字数组，反而比传原始的 UTF-8 字符串更耗费网络带宽！** 除非厂商提供底层的 gRPC/Protobuf 接口（将 ID 压缩为二进制的 int32），否则传 ID 在网络层毫无优势。
2) 致命缺陷：API 强耦合与版本灾难 (Version Hell)
这是架构师最怕的事情。如果 API 强制接收 ID，就意味着**客户端和服务端必须严格使用完全一致的词表版本**。
	- 假设你今天在后端写死了使用 GPT-4 的旧词表（`cl100k_base`）。
	- 明天 OpenAI 悄悄对模型底层进行了升级，换成了更高效的新词表（`o200k_base`，就像最近 GPT-4o 做的那样）。
	- 如果传的是文本，OpenAI 服务端自己换个新 Tokenizer 重新切分就行，对你的代码 **100% 透明，零感知**。
	- 如果你传的是 ID，你发过去的旧 ID 在新模型的脑子里对应的可能就变成了毫不相干的字，模型会立刻开始胡言乱语。**传文本，实现了客户端和服务端的彻底解耦。**
3) 安全与风控 (Safety & Prompt Injection)
	作为厂商（如阿里云、OpenAI），绝不敢直接把底层 ID 暴露给外部并允许直接注入。
	- 所有的风控系统（鉴黄、反政冶敏感、反暴力）都是**基于自然语言文本**进行正则匹配或小模型扫描的。
	- 如果允许黑客直接发送极其罕见或构造出的恶意 Token ID 序列，可以直接绕过安全网关的文本审查，甚至触发模型底层的越界漏洞。
4)  开发者体验 (DX)
	如果 API 只接收 ID，这就意味着每一个想用大模型的开发者，不管他是用 PHP、Ruby 还是 Shell 脚本，都必须先在本地费劲地实现一套 BPE 分词算法、下载几十 MB 的词表才能发请求。这会把 90% 的开发者挡在门外。


### 上下文窗口：即大模型的”短期记忆“
![[./assets/什么是大模型/file-20260504082605266.jpg]]

#### 为什么需要上下文
Web开发中，HTTP协议是无状态的，大模型的API也是无状态的。
当与大模型聊天时，模型不存在数据库记录聊天内容，但是为什么可以根据上下文来回答，是因为AI应用将当前问题及过去回答”**打包**“ 发送给大模型，这个每次必须携带的数据包大小就是上下文窗口。

#### 上下文的内容
- System Prompt（系统提示词）：设定 AI 的人设和底层规则（比如：“你是一个资深xxx，请用专业的语气回答...”）。
- Chat History（历史对话）：用户和 AI 之前几个回合的问答记录。
- Context / RAG Docs：你的系统从向量数据库里搜出来的、用来辅助回答的参考资料。
- User Input：用户最新敲下的一句话。
这四部分加起来消耗的 Token 总数，**绝不能超过**该模型规定的上下文窗口上限（比如早期 GPT-3.5 是 4K Token，现在的 GPT-4o 可以达到 128K，Gemini 1.5 Pro 可以达到夸张的 200万 Token）。

#### 上下文大小
1. **常规/开源模型的主流限制（8K - 32K Tokens）**
    - **代表**：大部分开源的 Llama-3 (8K)、Qwen 基础版等。
    - **容量**：换算成中文，大约能塞入 **1 万到 4 万字**。这基本相当于一篇长篇论文或几十页的 PDF。
2. **现代商业模型的标准配置（128K Tokens）**
    - **代表**：GPT-4o、Claude 3 Sonnet/Opus 等。
    - **容量**：换算成中文，大约能塞入 **10 万到 15 万字**。这足以塞进一整本中等厚度的书或者一个中型项目的全部核心代码。
3. **超长上下文模型（1M - 2M Tokens）**
    - **代表**：Google Gemini 1.5 Pro（最高支持 2M）、部分专门优化的长文本模型（如 Kimi 200万字版）。
    - **容量**：可以一次性塞入 **几百万字**，甚至几小时的音视频素材。    

**💡 灵魂拷问：既然有 1M 甚至 2M 的超大模型，我们是不是直接把几十本说明书全塞给它就行了？** **绝对不行！** 在实际工程中，我们极少会把几十万字直接塞进 Prompt，原因有三个：
1. **极度的昂贵**：API 是按 Token 计费的。每次提问都带着几十万字的历史背景，即使问题只有一句话，你也要支付极其高昂的输入费用。
2. **延迟灾难**：几十万字的 Prefill（预填充计算）会让大模型的响应时间从几百毫秒飙升到十几秒甚至几十秒。
3. **“迷失在中间”效应 (Lost in the Middle)**：论文证明，当上下文极长时，大模型对文档开头和结尾的信息提取很准，但极度容易遗忘或忽略中间段落的信息。
这就是为什么面对海量文档，**RAG（检索增强生成）** 至今依然是企业级应用的核心方案。
#### 保存方式
- **Redis（最常用）**：把 `session_id` 作为 Key，对话历史的 JSON 数组作为 Value 存入 Redis。读写极快，非常适合大模型这种高频读取历史记录的场景。
	- **业界的标准架构模式：**
		1. **热数据（Redis 做会话状态）**：当用户正在和 Agent 对话时（活跃的 Session），最近 10-50 轮对话记录实时读写 Redis。为了防爆内存，会给这个 Key 设置一个 `TTL`（比如 2 小时或 24 小时无操作后自动过期删除）。
		2. **冷数据（MySQL/PostgreSQL 做持久化）**：你的后端需要写一个异步任务（比如结合消息队列 Kafka 或 Spring 事件机制）。每当 Redis 里新增一条对话，或者一个 Session 结束时，异步将这段聊天记录持久化到 MySQL/PG 数据库中。
		3. **冷启动**：如果用户一周后重新打开这个对话界面，Redis 里早就没数据了。系统会先去 MySQL 查出最近的 N 条历史记录，重新塞回 Redis 预热，然后再去调大模型。
- **关系型数据库 (MySQL/PostgreSQL)**：用于永久保存用户的聊天记录（Chat History 表）。
- **框架支持**：在 LangChain 或 Spring AI 中，都有现成的 `RedisChatMessageHistory` 或 `PostgresChatMessageHistory` 类，你可以通过依赖注入直接挂载使用。

#### 上下文加工
发送给大模型之前，需要对信息进行**数据编排与组装 (Prompt Assembling)**
这个过程就像 Web 开发中的“模板渲染，具体包含以下三个步骤：
**第一步：角色格式化 (Role Assignment)**
大模型的 API 不接收一团乱糟糟的纯文本，它要求数据必须是结构化的 JSON 数组，并且每个文本块都要明确“是谁说的”。你需要将数据映射为 API 规定的角色：
- `"role": "system"`（系统）：用来装载 System Prompt 和 RAG 检索出来的文档。
- `"role": "user"`（用户）：用来装载用户以前的问题和当前的新问题。
- `"role": "assistant"`（AI）：用来装载 AI 之前的历史回复。

**第二步：模板注入 (Template Injection)**
对于 RAG 数据，你不能直接把文档扔进去，需要通过**提示词模板 (Prompt Template)** 进行引导。
- _你的后端代码会这样拼接_：“你是一个客服。请参考以下文档：【插入RAG数据】。来回答用户的问题：【插入用户当前提问】。”

**第三步：Token 预算检查 (Pre-flight Token Count)**
这是最关键的工程步骤。在发送 HTTP 请求前，你的后端需要调用本地的 Tokenizer 库（如 Python 的 `tiktoken`），提前计算组装好的 JSON 会消耗多少 Token。
- 如果小于窗口限制 $\rightarrow$ 直接发送。
- 如果大于窗口限制 $\rightarrow$ 触发拦截，执行“截断”或“丢弃”逻辑（这就引出了你的第三个问题）。

#### 上下文超限
当上下文超限时，可以采用 **“记忆压缩（Memory Compression）”** 或 **“摘要记忆（Summary Memory）”**。相比于简单的 FIFO（先进先出，直接把老聊天记录粗暴删掉），我们有更优雅的工程解法：
##### 方案 A：滚动摘要 (Rolling Window Summarization)
当对话历史即将触及 Token 上限时，你的后端程序在后台**偷偷发起一次额外的大模型调用**。
- **动作**：把你准备丢弃的那 10 轮老对话发给大模型，附带一个指令：“请把这段对话浓缩成 200 字以内的摘要，保留核心事实、用户偏好和关键结论。”
- **结果**：你把几千个 Token 的冗长对话，压缩成了一个几百 Token 的 `System Prompt` 附加项（例如：“历史背景：用户正在排查一个 K8s 支付服务 OOM 问题，已经排除了数据库死锁的可能”）。
- **优势**：极大节省了 Token，且模型依然“记得”前面的大体剧情。LangChain 中的 `ConversationSummaryMemory` 组件就是干这个的。
##### 方案 B：结合向量数据库的长效记忆 (Vector Store Backed Memory)
这是一种更降维打击的做法，直接把“聊天记录”当成“RAG 知识库”来用。
- **动作**：用户每说一句话，AI 每回一句话，你的后端都把这句话进行 Embedding（向量化），存入向量数据库（如 Milvus）。
- **组装 API 时的逻辑**：上下文窗口里只保留**最近的 3-5 轮**对话（极其短的 FIFO）。然后，拿着用户当前的新问题，去向量数据库里进行语义搜索，把**几个月前最相关的 2 轮老对话**搜出来，当作 RAG 资料塞进窗口。
- **场景**：比如你两周前告诉 AI“我的狗叫 Max”，今天你问“我狗叫啥”，系统能直接跨越时间的鸿沟，把两周前的那句话精准提取出来塞给大模型。

```json
    {
      "role": "user",
      "content": "那怎么修改这个内存限制？给出 kubectl 命令。"
    }
  ]
}
```



#### 大模型可以干什么

##### 1.听得懂人话,精准理解意图

你不用写复杂的代码指令,不用抠严谨的格式,只用日常大白话讲需求,它就能精准get到你想干嘛。
举个例子,你说「帮我把这段生硬的概念,改成小白能听懂的口语化教程」,它能立刻明白你的要求,你不需要
定义什么叫＂生硬＂、什么叫＂小白＂、什么叫＂口语化＂,它自己就能理解。这是所有AI应用能运转的基础。
##### 2.会逻辑思考,能拆解任务、做决策
给它一个明确的目标,它能拆解出一步步的执行步骤,还能判断「下一步该做什么」。
举个例子,输入「帮我规划一场周末的短途旅行」,它能拆解出查天气、定路线、找住宿、做预算这些步骤,并按
顺序推进。这种拆解任务和做决策的能力,就是我们后面Agent的「决策大脑」。
##### 3.能生成符合要求的各类内容
写文案、写代码、写总结、回答问题、梳理逻辑,只要你给它明确的要求和参考信息,它就能生成贴合需求的内
容。
这是它最终给用户＂交付结果＂的核心能力。
##### 4.能看懂规则,严格按要求执行
你给它定好明确的规矩,比如「必须严格按照我给的参考资料回答,不许自己瞎编」,或者「必须按照我给的固定
格式,告诉我要调用哪个工具」,它就能严格遵守,不会乱发挥。
这个能力是我们后面学 Function Calling、RAG的核心前提,如果模型不能严格按规则执行,整个系统就会乱套。

#### 大模型不能干什么
##### 1.没有执行能力，不能和外部世界交互
这是最核心的短板。大模型只有“嘴”,没有＂手＂。你让它帮你读电脑里的文件、查今天的实时天气、给客户发一封
邮件、在数据库里查一条记录,它只能给你写步骤,没法自己去做这些事。它也没法主动连接数据库、调用第三方
API、浏览网页,或者和外部的任何系统交互,只能在自己的＂脑子里＂转。
这就是我们后面要学Agent + Tool 工具调用、Function Calling和 MCP协议的核心原因:给大模型配上能动
手的工具,打通它和真实世界之间的通道,让它从只会出主意,变成能干活。
##### 2.知识有＂保质期＂，还容易瞎编
大模型的知识库是在某个时间点截止的,训练截止日期之后发生的事,它不知道。而且你公司内部的产品手册、专
属的制度文档、你自己写的私人笔记,它根本没见过,也不知道。
更麻烦的是,遇到它不知道的内容,它不一定会说＂我不知道＂,而是很可能一本正经地给你编一堆看起来很专业、
实际全是错的内容。行业里管这个叫「幻觉」(Hallucination)。
这就是我们后面要学RAG(检索增强生成)的核心原因:把最新的、专属的知识存起来,让模型回答的时候能调
取正确的参考资料,而不是靠＂记忆＂瞎猜。
##### 3.上下文窗口有上限,记不住太长的内容
白板就这么大,你把一本几百上千页的电子书、产品手册扔给它,,它根本记不住,会读了后面忘
前面,回答错漏百出。
这也是RAG要解决的问题:不把全部内容塞进窗口,而是按需检索最相关的片段。


#### 大模型原理
##### 第一阶段：数据的准备（从字符串到多维矩阵）
- **Tokenizer（切词与 ID 化）** 输入“Java 是一门面向” 会被Tokenizer根据词表切分为类似 `[Java, 是, 一门, 面向]` 这 4 个 Token，并映射为字典表里的整型数组 `[452, 9, 1204, 882]`。
- **Embedding（词向量化：从 ID 到特征空间）** 这是非常关键的一步。模型会拿着这些 ID，去查一张极其巨大的表（[[Embedding Matrix]]）。每个 ID 都会被替换成一个拥有几千个维度（比如 4096 维）的浮点数数组（向量）。
    - _后端视角_：这就像对数据做了一次带有语义的哈希（Semantic Hashing）。在这个 4096 维的空间里，代表“苹果”的向量和代表“梨”的向量距离非常近，而和“拖拉机”的距离非常远。
- **Positional Encoding（位置编码：打上时间戳）** 由于 Transformer 是**高度并行计算**的（不像早期的 RNN 要一个词一个词地读），它会瞬间吞下所有的向量。为了不让模型搞混“猫咬狗”和“狗咬猫”，系统会在每个词的向量上，强行叠加一个表示顺序的正弦/余弦数学信号。
    - _后端视角_：这就好比你在向消息队列（Kafka）并发投递消息时，给每一个 Payload 都打上了一个绝对有序的 `Sequence_ID`。
 ![[./assets/什么是大模型/file-20260509064437735.jpg]]
##### 第二阶段：Transformer 核心（自我注意力与知识计算）
这是大模型产生“理解力”的灵魂所在。几十层甚至上百层的 Transformer Block 会反复对这些向量进行蹂躏和重塑。每一层主要包含两个核心组件：
###### 1. Self-Attention 机制（建立上下文关联）
假如输入里有一个词叫“**苹果**”。模型怎么知道这是指“苹果公司”还是“吃个苹果”？这就靠 Self-Attention（自注意力机制）。
它为每个词生成三个专属的新向量：
- **Q (Query / 查询向量)**：我在寻找什么样的词来补充我的意思？
- **K (Key / 键向量)**：我能提供什么特征供别人查询？
- **V (Value / 值向量)**：我本身包含的实际内容是什么？
- _数据库类比_：这就相当于在一段文本内部做了一次高维的 `SQL JOIN`。所有的 Q 会去和所有的 K 进行点积计算（计算相关性分数）。如果“苹果”的 Q 发现旁边“手机”的 K 分数极高，它就会把“手机”的 V（信息）吸收到自己身上。经过这步操作后，“苹果”这个词的向量就被“污染”了，带上了“科技、手机”的语义。
这个过程用严谨的数学公式表示就是（这也是整个深度学习界最著名的公式之一）：
$$ Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}})V $$
$d_k$ 指的是 Key 向量（以及 Query 向量）的维度 (Dimension of Key)
###### 2. Feed-Forward Network (前馈神经网络：激活世界知识)
注意力机制只是让词与词之间建立了联系，但模型脑子里那些“秦始皇是哪年登基的”这种硬知识存在哪？
答案就存在前馈神经网络的权重矩阵里。
每个带上了上下文语境的词向量，会被送入这个极其庞大的多层感知机（MLP）。经过非线性激活函数（如 ReLU 或 SwiGLU）的过滤，激活模型在预训练时记忆的几十 TB 数据中的相关常识。
##### 第三阶段：概率输出（投骰子）
经过几十层 Transformer 的反复提纯，最后一个 Token（也就是“面向”）的向量已经包含了前面所有词的精华。
1. **Linear 层与 [[Softmax]]** 系统把这个终极向量送入一个巨大的全连接层，将其映射回字典表的大小（比如 10 万词）。然后通过一个 `Softmax` 算法，将这些数字转换成一个总和为 100% 的概率分布。
2. **采样（Sampling）** 模型可能会输出这样的概率：
    - `对象`：95%
    - `过程`：4%
    - `工资`：0.1% 这就是为什么大模型的 API 里通常会有一个 `Temperature`（温度）参数。如果温度设为 0，模型就死板地永远选概率最高的“对象”；如果温度设高，它就有可能偶尔选“过程”，产生一种“随机的创造力”。

## 面试题

### 一、 大模型基础概念与 Token 机制

**1. 什么是大模型（LLM）？它的训练过程本质上是在做什么？**
- **参考答案**：大模型是指参数量巨大且训练数据海量的语言模型。参数可以理解为模型内部无数个控制电流的“小推子和旋钮”，代表其记住了海量知识的精华。大模型的训练过程，本质上是一个数学优化过程：通过不断试错和统计计算（损失函数与前向传播），寻找一组“最优参数”（梯度下降与反向传播），从而使得模型预测的误差降到最低。
    

**2. 什么是 Token？它是如何被切分的？**
- **参考答案**：Token 是大模型处理文本的基本单元，可以理解为文字被切碎后的小碎片，既不是字也不是词。它的切分依据主要是通过统计算法（如 BPE 算法）生成词表，核心思想是把经常紧挨着出现的字符合并成新的大字符。完整流程是：Tokenizer 切分文本 -> 映射为 Token ID (数字) -> 模型计算 -> 输出新 ID -> 解码回人类文本。
    

**3. 为什么在调用大模型 API 时，建议后端 Agent 直接传输文本，而不是在本地转成 Token ID 后传给模型？**
- **参考答案**：
    - **带宽消耗**：在基于 JSON 的 RESTful API 中，将纯文本转换为一长串数字数组进行序列化，反而比直接传输 UTF-8 字符串更耗费网络带宽。
    - **版本灾难与解耦**：传 ID 会导致客户端与服务端强耦合，若服务端悄悄升级了词表（如 OpenAI 更换新 Tokenizer），旧 ID 会导致模型胡言乱语；传纯文本则对代码 100% 透明，实现彻底解耦。
    - **安全与风控**：所有的风控系统通常基于自然语言文本进行扫描，允许直接传 ID 可能绕过安全审查，甚至触发底层越界漏洞。
    - **开发者体验**：强制传 ID 需要开发者在本地实现复杂的 BPE 算法并下载词表，极大地增加了接入成本。
        

**4. 随着上下文 Token 数量的增加，会对 GPU 产生什么影响？业界是如何优化的？**
- **参考答案**：为了结合上文预测下一个词，模型会在每层神经网络中把 Token 的“特征状态”缓存下来，这个缓存区称为 KV Cache。随着 Token 数量增加，KV Cache 占用的 GPU 显存会呈**线性爆炸**。例如 128K Token 可能需要消耗十几 GB 的显存。因此，业界 MLOps 正在发展 `vLLM`、`PagedAttention` 等技术，像操作系统管理虚拟内存一样去优化显存。

### 二、 上下文窗口与工程架构
**5. 什么是上下文窗口？由于 API 是无状态的，我们在工程上通常如何管理聊天历史（上下文）？**
- **参考答案**：上下文窗口是大模型的“短期记忆”，指每次 API 调用时必须打包携带的数据包总大小，通常包含系统提示词、历史对话、RAG 文档和用户输入。
    - 工程上通常采用 **Redis** 管理热数据，将 `session_id` 作为 Key，对话历史的 JSON 数组作为 Value 进行高频读写，并设置过期时间防爆内存。
    - 对于冷数据，会通过异步任务（如 Kafka/Spring 事件）将结束的会话持久化到 MySQL 或 PostgreSQL 中。
    - 冷启动时，会从数据库查出历史记录重新预热到 Redis 中。
        
**6. 当用户的对话历史超出了上下文窗口上限，除了粗暴的 FIFO（先进先出）截断，还有哪些优雅的工程解法？**
- **参考答案**：
    - **滚动摘要 (Rolling Window Summarization)**：在后台发起额外调用，将准备丢弃的老对话浓缩成简短摘要（保留核心事实和结论），作为 `System Prompt` 的附加项塞入窗口。
    - **结合向量数据库的长效记忆**：将所有的历史对话向量化后存入向量数据库（如 Milvus）。每次请求时，窗口只保留极其短的最近几轮对话，同时拿着当前问题去数据库中进行语义搜索，将历史中最相关的老对话提取出来当作 RAG 资料塞给模型。
### 三、 大模型能力边界与底层原理
**7. 大模型有哪些核心短板（不能干什么）？针对这些短板对应的解决方案是什么？**
- **参考答案**：
    - **无执行能力**：大模型没有“手”，无法主动连接数据库或调用外部 API。解决方案是使用 Agent + Tool 工具调用、Function Calling 或 MCP 协议，让它能和真实世界交互。
    - **知识有保质期且容易幻觉**：大模型不知道训练截止日期之后的事情，也不知道企业私有数据，遇到盲区容易一本正经地瞎编。解决方案是引入 RAG（检索增强生成），通过按需检索参考资料来限制其瞎猜。
    - **上下文上限**：记不住过长的内容，塞入整本书会导致读了后面忘前面。同样依赖 RAG 按需检索最相关的片段。

**8. 简述大模型处理一段文本并输出回答的完整底层计算链路。**
- **参考答案**：
    - **第一阶段：数据准备**。Tokenizer 将文本切分为 ID，随后通过 Embedding 层查表，将 ID 映射为高维浮点数向量（赋予语义），并叠加 Positional Encoding（位置编码）打上顺序时间戳。
    - **第二阶段：Transformer 计算**。包含 Self-Attention（自注意力机制）和 Feed-Forward Network（前馈神经网络）。前者通过计算 Q、K、V 向量建立词与词之间的上下文关联；后者则是一个多层感知机，负责激活模型预训练时记忆的硬知识。
    - **第三阶段：概率输出**。最后的向量通过 Linear 层与 Softmax 算法映射回字典表概率分布，并结合 Temperature（温度）参数进行采样，投骰子决定最终输出的词。

**9. 大模型是如何理解词与词之间的联系的（如区分“吃个苹果”和“苹果公司”）？请写出相关的核心数学公式。**
- **参考答案**：大模型通过 **Self-Attention（自注意力机制）** 来建立上下文关联。它会为每个词生成三个专属向量：Query (Q，查询)、Key (K，特征) 和 Value (V，内容)。模型通过计算 Q 和其他词的 K 之间的相关性分数，决定吸收哪些词的 V 信息，从而让词向量带上正确的上下文语境。 其核心数学公式为：
    $$ Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}})V $$
